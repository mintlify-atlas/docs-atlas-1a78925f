## Overview

FleetGuard360 uses a GraphQL API for all data operations. The API provides type-safe queries and mutations for managing alerts, priorities, and related entities.

## API Configuration

### Endpoint Setup

The GraphQL API is configured with a relative endpoint that works with the Vite proxy:

```typescript
const API_URL = "/api";
```

### Proxy Configuration

In development, requests are proxied to the backend server. From `vite.config.ts`:

```typescript
proxy: {
  "/api": {
    target: "http://fleetguard-api.ddns.net",
    changeOrigin: true,
    rewrite: (path) => path.replace(/^/api/, ""),
    configure: (proxy, _options) => {
      proxy.on("proxyReq", (proxyReq, _req, _res) => {
        proxyReq.setHeader("Origin", "http://fleetguard-api.ddns.net");
      });
    },
  },
}
```

<Note>
  The `/api` prefix is stripped when forwarding to the backend server.
</Note>

## Authentication

### JWT Bearer Tokens

All GraphQL requests require authentication using JWT Bearer tokens:

```typescript
const token = localStorage.getItem("jwt_token") || sessionStorage.getItem("jwt_token");

const response = await fetch(`${API_URL}${endpoint}`, {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    Authorization: `Bearer ${token}`,
  },
  body: JSON.stringify({ query, variables }),
});
```

### Session Expiration Handling

The API automatically handles expired sessions:

```typescript
if (response.status === 401 || response.status === 403) {
  localStorage.removeItem("jwt_token");
  sessionStorage.removeItem("jwt_token");
  navigate("/", { state: { sessionExpired: true } });
  throw new Error("Session expired");
}
```

<Warning>
  When a 401 or 403 status is received, tokens are cleared and the user is redirected to the login page.
</Warning>

## Request/Response Format

### GraphQL Response Interface

```typescript
interface GraphQLResponse<T> {
  data?: T;
  errors?: Array<{ message: string }>;
}
```

### Making Requests

The `useGraphQL` hook provides a type-safe wrapper:

```typescript
const { fetchGraphQL } = useGraphQL();

const result = await fetchGraphQL<ResponseType>(
  "/alerts/graphql",
  query,
  variables // optional
);
```

## API Endpoints

FleetGuard360 uses multiple GraphQL endpoints for different domains:

- `/api/alerts/graphql` - Alert type configuration
- `/api/panel/graphql` - Alert instances and history

## Error Handling

The GraphQL client includes comprehensive error handling:

```typescript
const result: GraphQLResponse<T> = await response.json();

if (result.errors) {
  throw new Error(result.errors[0]?.message || "GraphQL error");
}

if (!result.data) {
  throw new Error("No data returned from GraphQL");
}

return result.data;
```

### Common Error Scenarios

<ParamField path="401/403" type="error">
  **Session Expired** - Token is invalid or expired. User is redirected to login.
</ParamField>

<ParamField path="GraphQL Errors" type="error">
  **Validation/Business Logic Errors** - Returned in the `errors` array with descriptive messages.
</ParamField>

<ParamField path="No Data" type="error">
  **Empty Response** - Query succeeded but returned no data.
</ParamField>

## Best Practices

<Note>
  **Type Safety** - Always provide TypeScript types when calling `fetchGraphQL` for compile-time type checking.
</Note>

<Note>
  **Token Storage** - Tokens are stored in both `localStorage` (persistent) and `sessionStorage` (session-only) for flexibility.
</Note>

<Warning>
  **Error Handling** - Always wrap GraphQL calls in try-catch blocks to handle network and API errors gracefully.
</Warning>

<Note>
  **Auto-redirect** - The hook automatically redirects to login on authentication errors - no manual handling needed.
</Note>
