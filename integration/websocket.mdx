## Overview

FleetGuard360 uses WebSocket connections to deliver real-time alert notifications. The system uses **SockJS** for cross-browser WebSocket support and **STOMP** (Simple Text Oriented Messaging Protocol) for message handling.

## Connection Setup

### WebSocket Endpoint

The WebSocket connection uses a relative path that works with the Vite proxy:

```typescript
const socket = new SockJS("/notifications/ws-alerts");
const stompClient = Stomp.over(socket);
```

### Proxy Configuration

From `vite.config.ts`, WebSocket requests are proxied to the backend:

```typescript
proxy: {
  "/notifications": {
    target: "http://fleetguard-api.ddns.net",
    changeOrigin: true,
    ws: true, // Important: enables WebSocket proxying
    secure: false,
  },
}
```

<Warning>
  The `ws: true` option is critical for WebSocket proxy support.
</Warning>

## Using the Hook

### Basic Usage

The `useWebSocketNotifications` hook handles all WebSocket logic:

```typescript
import { useWebSocketNotifications } from "@/hooks/use-websocket-notifications";

const { isConnected, notifications, connect, disconnect, clearNotifications } = 
  useWebSocketNotifications({
    onNotification: (notification) => {
      console.log("New alert:", notification);
    },
    autoConnect: true, // Optional, defaults to true
  });
```

### Hook Options

<ParamField path="onNotification" type="function" optional>
  Callback function called when a new notification arrives.
</ParamField>

<ParamField path="autoConnect" type="boolean" default="true">
  Whether to automatically connect on mount. Set to `false` if you want to control connection manually.
</ParamField>

### Return Values

<ParamField path="isConnected" type="boolean">
  Current connection status. `true` when connected, `false` otherwise.
</ParamField>

<ParamField path="notifications" type="AlertNotification[]">
  Array of received notifications (newest first).
</ParamField>

<ParamField path="connect" type="function">
  Manually initiate WebSocket connection.
</ParamField>

<ParamField path="disconnect" type="function">
  Manually close WebSocket connection.
</ParamField>

<ParamField path="clearNotifications" type="function">
  Clear all stored notifications.
</ParamField>

## Connection Lifecycle

### Connection Establishment

From `use-websocket-notifications.ts:32-46`:

```typescript
const connect = () => {
  const socket = new SockJS("/notifications/ws-alerts");
  const stompClient = Stomp.over(socket);

  // Disable debug logs (optional)
  stompClient.debug = () => {};

  stompClient.connect(
    {},
    (frame) => {
      console.log("WebSocket conectado:", frame);
      setIsConnected(true);

      // Subscribe to alerts channel
      stompClient.subscribe("/all/alerts", (message) => {
        // Handle incoming messages
      });
    },
    (error) => {
      console.error("Error en WebSocket:", error);
      setIsConnected(false);
    }
  );
};
```

### Subscription Topic

Notifications are published to the `/all/alerts` topic:

```typescript
stompClient.subscribe("/all/alerts", (message) => {
  const alert: AlertNotification = JSON.parse(message.body);
  console.log("Notificación recibida:", alert);
});
```

<Note>
  The topic name `/all/alerts` is configured on the backend STOMP message broker.
</Note>

## Message Handling

### Parsing Messages

From `use-websocket-notifications.ts:50-66`:

```typescript
stompClient.subscribe("/all/alerts", (message) => {
  try {
    console.log("Mensaje RAW recibido:", message.body);
    const alert: AlertNotification = JSON.parse(message.body);
    console.log("Notificación parseada:", alert);

    // Add to notifications array (newest first)
    setNotifications((prev) => [alert, ...prev]);

    // Trigger callback if provided
    if (onNotification) {
      onNotification(alert);
    }
  } catch (error) {
    console.error("Error procesando notificación:", error);
  }
});
```

### State Management

### Automatic Reconnection

```typescript
stompClient.connect(
  {},
  (frame) => { /* ... */ },
  (error) => {
    console.error("Error en WebSocket:", error);
    setIsConnected(false);

    // Attempt to reconnect after 5 seconds

      console.log("Intentando reconectar...");
      connect();
    }, 5000);
  }

```

<Warning>
  **Exponential Backoff** - The current implementation uses a fixed 5-second delay. Consider implementing exponential backoff for production.
</Warning>

## Disconnection

### Manual Disconnect

From `use-websocket-notifications.ts:87-94`:

```typescript
const disconnect = () => {
  if (stompClientRef.current?.connected) {
    stompClientRef.current.disconnect(() => {
      console.log("WebSocket desconectado");
      setIsConnected(false);
    });
  }
};
```

### Auto-disconnect on Unmount

```typescript
useEffect(() => {
  if (autoConnect) {
    connect();
  }

  return () => {
    disconnect();
  };
}, []);
```

<Note>
  The connection is automatically closed when the component unmounts to prevent memory leaks.
</Note>

## Best Practices

<Note>
  **Auto-connect by default** - Set `autoConnect: true` for components that should always receive notifications.
</Note>

<Note>
  **Manual control** - Use `autoConnect: false` and call `connect()` manually when you need precise control over connection timing.
</Note>

<Warning>
  **Clean up on unmount** - The hook automatically disconnects on unmount, but if you're using `autoConnect: false`, ensure you call `disconnect()` when done.
</Warning>

<Note>
  **Connection state** - Always check `isConnected` before assuming notifications are being received.
</Note>

<Note>
  **Debug mode** - Enable STOMP debug logs during development by removing the `stompClient.debug = () => {}` line.
</Note>

## Troubleshooting

### Connection Issues

1. **Check proxy configuration** - Ensure `ws: true` is set in `vite.config.ts`
2. **Verify endpoint** - The backend must serve WebSocket at `/notifications/ws-alerts`
3. **CORS settings** - Backend must allow WebSocket connections from your origin

### Missing Notifications

1. **Check subscription topic** - Verify `/all/alerts` is the correct topic
2. **Inspect connection state** - Use `isConnected` to monitor connection status
3. **Enable debug logs** - Remove `stompClient.debug = () => {}` to see STOMP messages

### Performance

1. **Limit notification history** - Consider trimming the `notifications` array to prevent memory issues:
   ```typescript
   setNotifications((prev) => [alert, ...prev].slice(0, 50));
   ```
2. **Throttle toast notifications** - Prevent notification fatigue by throttling toast displays
