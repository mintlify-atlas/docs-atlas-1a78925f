## Project Structure

The application follows a modular, feature-based architecture:

```
src/
├── pages/              # Route-level page components
│   ├── Login.tsx       # Authentication page
│   ├── Dashboard.tsx   # Main dashboard view
│   ├── PanelDeAlertas.tsx  # Alerts management
│   ├── Estadisticas.tsx    # Statistics and analytics
│   └── NotFound.tsx    # 404 error page
├── components/         # Reusable UI components
│   ├── ui/            # shadcn/ui components
│   ├── Sidebar.tsx    # Navigation sidebar
│   └── NotificationBell.tsx  # Real-time notifications
├── hooks/             # Custom React hooks
│   ├── use-graphql.ts           # GraphQL API integration
│   ├── use-websocket-notifications.ts  # WebSocket hook
│   ├── use-toast.ts             # Toast notifications
│   └── use-mobile.tsx           # Mobile detection

│   └── utils.ts       # Helper utilities (clsx, cn)

```

## Application Architecture

### Routing Layer

The application uses React Router DOM for client-side routing. All routes are defined centrally in `App.tsx:20-26`:

```tsx
<BrowserRouter>
  <Routes>
    <Route path="/" element={<Login />} />
    <Route path="/dashboard" element={<Dashboard />} />
    <Route path="/panel-de-alertas" element={<PanelDeAlertas />} />
    <Route path="/estadisticas" element={<Estadisticas />} />

</BrowserRouter>
```

## API Integration Layer

### GraphQL Communication

The `useGraphQL` hook (`hooks/use-graphql.ts`) provides a standardized interface for GraphQL API calls:

<Steps>
  <Step title="Authentication">
    Automatically includes JWT token from storage in all requests (`use-graphql.ts:18`)
  </Step>
  
  <Step title="Request Execution">
    Sends POST requests to `/api` endpoint with GraphQL query and variables (`use-graphql.ts:25-32`)
  </Step>
  
  <Step title="Error Handling">
    Handles 401/403 responses by clearing auth tokens and redirecting to login (`use-graphql.ts:34-39`)
  </Step>
  
  <Step title="Response Processing">
    Parses GraphQL response and extracts data or throws errors (`use-graphql.ts:41-51`)
  </Step>
</Steps>

**Example Usage:**

```tsx
const { fetchGraphQL } = useGraphQL();

const data = await fetchGraphQL<ResponseType>(
  '/graphql',
  `query { alerts { id, type, priority } }`,
  { limit: 10 }
);
```

### WebSocket Integration

Real-time notifications are handled via WebSocket using the `useWebSocketNotifications` hook (`hooks/use-websocket-notifications.ts`):

**Architecture:**
- **SockJS** (`use-websocket-notifications.ts:35`) - Provides WebSocket with fallback transports
- **STOMP** (`use-websocket-notifications.ts:36`) - Messaging protocol over WebSocket
- **Auto-reconnect** (`use-websocket-notifications.ts:73-76`) - Automatic reconnection on disconnect

**Connection Flow:**

```tsx
// 1. Initialize WebSocket connection
const socket = new SockJS("/notifications/ws-alerts");
const stompClient = Stomp.over(socket);

// 2. Subscribe to alerts channel
stompClient.subscribe("/all/alerts", (message) => {
  const alert = JSON.parse(message.body);
  // Handle notification
});
```

<Note>
The WebSocket connection uses relative paths (`/notifications/ws-alerts`) to work seamlessly with both development proxy and production Vercel rewrites.
</Note>

## Authentication Flow

<Steps>
  <Step title="Login">
    User enters credentials on `/` (Login page)
  </Step>
  
  <Step title="Token Storage">
    JWT token is stored in localStorage or sessionStorage based on "Remember me" option
  </Step>
  
  <Step title="Protected Navigation">
    User navigates to protected routes (`/dashboard`, `/panel-de-alertas`, `/estadisticas`)
  </Step>
  
  <Step title="Token Validation">
    Each API request includes the token via `useGraphQL` hook
  </Step>
  
  <Step title="Session Expiry">
    If token is invalid (401/403), user is redirected to login and tokens are cleared
  </Step>
</Steps>

## Development Workflow

### Hot Module Replacement (HMR)

Vite provides instant HMR powered by:
- **@vitejs/plugin-react-swc** - Ultra-fast React refresh using SWC compiler
- **Component Tagger** (dev mode) - Adds metadata for debugging

### Development Server

Configured in `vite.config.ts:8-28`:

```typescript
server: {
  host: "::",        // IPv6 support
  port: 3000,
  proxy: {
    "/api": {        // GraphQL API proxy
      target: "http://fleetguard-api.ddns.net",
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api/, "")
    },
    "/notifications": {  // WebSocket proxy
      target: "http://fleetguard-api.ddns.net",
      changeOrigin: true,
      ws: true           // Enable WebSocket proxying
    }
  }
}
```

<Warning>
The proxy configuration is only active during development (`npm run dev`). Production uses Vercel rewrites defined in `vercel.json`.
</Warning>

## Best Practices

<AccordionGroup>
  <Accordion title="Component Design">
    - Keep components small and focused
    - Use composition over inheritance
    - Leverage shadcn/ui components as building blocks
    - Extract reusable logic into custom hooks
  </Accordion>
  
  <Accordion title="State Management">
    - Use TanStack Query for all server data
    - Keep component state local when possible
    - Lift state only when necessary
    - Avoid prop drilling with context when appropriate
  </Accordion>
  
  <Accordion title="TypeScript Usage">
    - Define interfaces for API responses
    - Use type inference when possible
    - Leverage path aliases for clean imports
    - Type custom hooks return values
  </Accordion>
  
  <Accordion title="Performance">
    - Lazy load routes with React.lazy()
    - Memoize expensive calculations
    - Use React.memo() for pure components
    - Optimize WebSocket subscriptions
  </Accordion>
</AccordionGroup>
